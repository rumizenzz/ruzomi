# Polymarket Docs Benchmark

Reference date: March 8, 2026

Official sources reviewed:

- `https://docs.polymarket.com/`
- `https://docs.polymarket.com/polymarket-learn/get-started/making-your-first-trade`

## Structural findings

- Polymarket keeps its developer docs as a dedicated product surface, not a subsection of the marketing site.
- The docs landing is builder-first. The first frame pushes quickstart, core concepts, API reference, SDKs, builder program, help desk, and status.
- Code samples are above the fold on the overview page, which makes the docs feel active and implementation-ready.
- The docs navigation is organized around concepts, APIs, SDKs, rewards, and support instead of broad brand storytelling.
- Operational destinations such as help desk and status are visible inside the docs IA, which makes the documentation feel like part of a living production system.

## PayToCommit takeaways

- `docs.paytocommit.com` should feel like a real operating surface with search, left navigation, API reference, and support links visible immediately.
- The docs home should start with real implementation primitives: pool browse, pool detail, access grants, fees and payouts, Commitment Network events, and rules standards.
- Reward documentation should live beside API and payout documentation so the product economics read as part of the system, not a separate promo program.
- Docs copy should stay literal and procedural. The reader should understand keys, consent, fees, rules, and ledger events without reading brand-positioning language.
